アプリケーション開発要件（要件定義）

この文書は、提示された「課題を起点に、人とAIと行動を繋ぐ基盤の構築」という構想に基づき、最小実行可能製品（MVP）を開発するための具体的な要件を定義します。

1\. 共通プロトコル（ビジネスロジックの中核）

**要件：** 以下の6ステップを処理するバックエンドロジック（API）を構築すること。

| ステップ | 機能概要 | データ項目（課題スキーマと連携） |
| ----- | :---: | :---: |
| **① 収集 (Intake)** | ユーザーからの課題情報（質問、自由記述）を収集し、初期の`Problem`データを作成する。 | `Problem.context`, `Problem.symptoms`, `Problem.constraints`など |
| **② 構造化 (Structuring)** | 収集データをタグ化、数値化し、スキーマに則ったデータ形式に変換する。 | `Problem.severity`, `Problem.frequency`, 内部`tags` |
| **③ 優先度決定 (Prioritizing)** | 現在の未解決課題の中から、優先度の高いもの（1つ）を選定する。 | `Problem.priority` (計算結果) |
| **④ 解決提案 (Solving)** | 優先度の高い課題に対し、具体的な解決策（`Action`）を提案する。代替案も含む。 | `Action`オブジェクト一式 |
| **⑤ 実行 (Execution)** | 提案された`Action`の実行を支援する（リンク提示、通知設定など）。 | `Action.type`, `Action.link`など |
| **⑥ フィードバック (Evaluating)** | 実行結果、満足度、不満を収集し、`Outcome`として記録する。 | `Outcome`オブジェクト一式 |

2\. データモデル要件（スキーマ定義）

**要件：** 以下の3つの主要なテーブル（またはコレクション）を定義し、永続化すること。

| テーブル名 | フィールド群 | 型・制約 | 補足 |
| ----- | :---: | :---: | :---: |
| **problems** | `context`, `symptoms`, `constraints`, `goal`, `severity`, `frequency`, `priority`, `status` (未解決/解決済) | JSON, String, Integer | 課題の定義 |
| **actions** | `problem_id` (外部キー), `type`, `cost` (時間/金), `expected_gain`, `risk`, `link`, `is_recommended` | JSON, String, Decimal | 推奨される行動 |
| **outcomes** | `action_id` (外部キー), `did_execute`, `delta` (改善量), `satisfaction`, `complaint`, `feedback_date` | Boolean, Decimal, Integer | 行動の結果と評価 |

**データ設計の原則：** コアなフィールドは共通とし、ドメイン固有の情報は柔軟な`tags`フィールド（JSONまたは配列）として保持する。3. エージェント機能要件（役割分担）

**要件：** 共通プロトコルの各ステップに対応するマイクロサービス的な役割を果たすバックエンドモジュール（または関数）を開発すること。

| エージェント名 | 担当プロトコルステップ | 主な入出力 |
| ----- | :---: | :---: |
| **Intake Agent** | ① 収集 | 入力: ユーザー質問 / 出力: 初期`Problem` |
| **Structuring Agent** | ② 構造化 | 入力: 初期`Problem` / 出力: 構造化`Problem`と`tags` |
| **Prioritizer** | ③ 優先度決定 | 入力: 全`Problem`s / 出力: 最優先`Problem` ID |
| **Solver** | ④ 解決提案 | 入力: 優先`Problem` / 出力: 1つ以上の`Action`s |
| **Executor** | ⑤ 実行支援 | 入力: `Action` / 出力: リンク、通知設定APIコール |
| **Evaluator** | ⑥ フィードバック | 入力: ユーザーフィードバック / 出力: `Outcome`データ |

4\. ユーザーインターフェース（Webアプリ）要件

**要件：** ユーザーが「考える→やる→振り返る」を完結できる以下の5つの画面（SPA/Webアプリ）を実装すること。

| 画面名 | 主な機能（対応エージェント） | 詳細要件 |
| ----- | :---: | :---: |
| **1\. ダッシュボード** | 優先度提示 (Prioritizer) | \- 現在の最重要課題（1つ）を大きく表示。- 推奨アクション（1つ）を表示し、実行画面へ誘導。 |
| **2\. 診断画面 (Intake)** | 課題収集 (Intake Agent) | \- 分岐する質問フロー（ウィザード形式）を実装。- 自由記述フィールドを設ける。- データ送信で`Problem`の初期化をトリガー。 |
| **3\. 結果画面 (Solver)** | 提案表示 (Solver) | \- 提案された`Action`の内容、リスク、期待効果を表示。- 提案の根拠（理由）と代替案を表示。 |
| **4\. 実行画面 (Executor)** | 行動支援 (Executor) | \- `Action`を実行するためのワンタップボタン（外部リンク遷移、通知設定など）。 |
| **5\. フィードバック画面 (Evaluator)** | 結果記録 (Evaluator) | \- 実行の有無、主観満足度、不満点を入力するフォーム。- `Outcome`データを永続化するAPIをコール。 |

5\. 技術・実装要件（MVPフェーズ）

| 要件カテゴリ | 詳細 | 補足 |
| ----- | :---: | :---: |
| **フレームワーク** | Next.js (フロントエンド、バックエンドAPI) | 迅速な開発とサーバーレスデプロイを考慮。 |
| **データベース** | Supabase (PostgreSQL) | リアルタイムDB機能と認証機能の活用を検討。 |
| **API連携** | フェーズ1: リンク提示 | 外部ストア、記事などへの静的リンク提示に限定。 |
| **AI/ML** | 初期はルールベース/シンプルなLLMプロンプト | 構造化、提案のロジックは段階的に高度化。 |
| **優先度計算** | `priority = impact × feasibility`のシンプルなロジックを実装。 | ユーザーからの入力や構造化データから算出。 |
