「課題を起点に、人とAIと行動を繋ぐ基盤の構築」プロジェクトにおけるIntake Agent（収集エージェント）を実装するための要件を以下にまとめます。

Intake Agentは、ユーザーの「考える負担を極限まで減らし、「実行できる」状態を作る」というシステムの根本的な存在意義を実現するための、最初の接点となるエージェントです。

## 1\. 役割と目的（共通プロトコルにおける位置づけ）

| 項目         | 詳細                                           | 参照エージェント     |
| :--------- | :------------------------------------------- | :----------- |
| **担当ステップ** | ① 収集 (Intake)                                | Intake Agent |
| **機能概要**   | ユーザーからの課題情報（質問、自由記述）を収集し、初期のProblemデータを作成する。 | N/A          |
| **目的**     | ユーザーに考えさせずに入力を極限まで減らし、課題を構造化するための一次情報を取得する。  | N/A          |

## 2\. 機能要件（最小プロダクト：ゲーム購入の後悔を減らすAI）

MVPにおけるIntake Agentの機能は、「診断（質問）」と「課題の初期化」に限定されます。

| 要件             | 詳細                                                        |
| :------------- | :-------------------------------------------------------- |
| **質問フローの実装**   | 分岐型の診断フロー（ウィザード形式）を実装する。質問数は5〜10問に限定し、ユーザーの好みやプレイ傾向を取得する。 |
| **入力方式**       | ユーザーの負荷を減らすため、基本は選択式（Yes/Noや選択）とし、補助機能として自由記述フィールドを設ける。   |
| **非常口機能**      | いつでもユーザーが「これ違う」と入力できる非常口機能を設計に組み込む。                       |
| **課題の初期化**     | 診断結果をデータとして送信し、バックエンドで`Problem`データの初期化をトリガーする。            |
| **パーソナライズの準備** | 収集したデータはユーザーIDと紐づけ、ユーザーごとにパーソナライズされた履歴表示の基盤とする。           |

## 3\. データ要件（収集するデータモデル）

Intake Agentが収集したデータは、主に以下のテーブルに格納されます。

| テーブル         | フィールド例                               | 補足                                       |
| :----------- | :----------------------------------- | :--------------------------------------- |
| **users**    | id, 年齢層（ざっくり）, 好みタグ                  | ユーザーの基本属性データ。                            |
| **answers**  | user\_id, question\_id, answer       | 診断で入力された回答ログ。                            |
| **problems** | context, symptoms, constraints, goal | 課題の定義情報。Intake Agentの出力として初期データが作成される。   |
| **tags**     | 不満タグ（時間、お金、ストレス、難易度など）               | 将来的な社会課題への接続を視野に入れ、初期から横展開可能なタグを設計に組み込む。 |

## 4\. UI/UX要件

| 画面名               | 詳細要件                                                                                   | UX設計思想                                        |
| :---------------- | :------------------------------------------------------------------------------------- | :-------------------------------------------- |
| **診断画面 (Intake)** | 分岐する質問フロー（ウィザード形式）を実装し、自由記述フィールドを設ける。                                                  | ユーザーを「質問する側」ではなく「選ぶ側」にし、「考えなくても始められる」状態を実現する。 |
| **アクションがない場合**    | 最適なアクションがない場合でも、前の画面に戻らずにチャット形式でテキストを入力し、次の診断プロセスに進めるようにする。                            | ユーザーが途中で離脱することを防ぎ、データ収集を継続する。                 |
| **課題の提示**         | 考える負担を減らすため、ユーザーが自ら課題を言わなくても、AIが自動的に課題や世の中の仕組み、最新情報について収集し、解決手段、優先順位などを自動で計画できる体制を整える。 | N/A                                           |

## 5\. 技術要件

| 要素            | 詳細                                     | 補足                                                     |
| :------------ | :------------------------------------- | :----------------------------------------------------- |
| **フレームワーク**   | Next.js (App Router)                   | 迅速なUI開発とバックエンドAPI（エージェント）のシームレスな統合。                    |
| **バックエンド/DB** | Supabase (PostgreSQL)                  | 認証機能を利用し、ユーザーごとに診断履歴がパーソナライズされるように実装する。                |
| **AIの利用**     | OpenAIなどのAPI利用（軽量モデル）                  | 初期段階では、質問生成の補助や、収集した自由記述（不満テキスト）の簡易分類のみに限定し、コストを最小化する。 |
| **ロジック**      | 初期はルールベース（if文）で判定ロジックを構築し、最短でMVPを実現する。 | N/A                                                    |
